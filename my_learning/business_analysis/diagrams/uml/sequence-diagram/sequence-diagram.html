Sequence Diagram

Sequence Diagram (Sơ đồ tuần tự) là một loại biểu đồ tương tác trong UML (Unified Modeling Language), tập trung vào việc mô tả:

  • Cách các đối tượng (objects) tương tác với nhau
  • Thứ tự thời gian (timeline) của các tương tác đó
  • Luồng message (thông điệp) giữa các thành phần

Nói ngắn gọn: Sequence Diagram = “Ai gọi ai, gọi khi nào, và gọi theo thứ tự nào”

Đây là công cụ thiết yếu trong phân tích và thiết kế phần mềm giúp trực quan hóa luồng xử lý của một kịch bản cụ thể.

Đặc điểm quan trọng:

  • Trục dọc: thời gian (top → bottom)
  • Trục ngang: các đối tượng / thành phần hệ thống

Vai trò & mục đích

Sequence Diagram dùng để:

  • Hiểu behavior (hành vi hệ thống): Xem hệ thống hoạt động ra sao theo thời gian
  • Thiết kế & phân tích:
    • Thiết kế logic backend/API
    • Phân tích use case: Là tài liệu trung gian quan trọng chuyển đổi từ yêu cầu người dùng (use case) sang sơ đồ lớp (class diagram) và các bước cài đặt hệ thống
  • Documentation:
    • Mô tả luồng xử lý
    • Hỗ trợ team dev hiểu hệ thống: Giúp mọi thành viên (lập trình viên, tester, quản lý) hiểu rõ luồng xử lý phức tạp mà không cần đọc sâu vào code
  • Debug & architecture:
    • Tìm lỗi logic: Việc mô phỏng sớm giúp nhóm nhận diện các thiếu sót trong logic nghiệp vụ hoặc luồng dữ liệu trước khi thực thi
    • Thiết kế microservices / API flow

Thành phần chính trong Sequence Diagram

  1. Actor
    Đại diện cho người dùng hoặc hệ thống bên ngoài.
    Ví dụ: User, Payment Gateway
  2. Lifeline (Đường sống)
    Là đối tượng tham gia vào interaction. Được biểu diễn bằng: Tên object ở trên, Đường thẳng dọc xuống.
    Ý nghĩa: Thể hiện object “tồn tại theo thời gian”.
  3. Activation (Execution)
    Thanh hình chữ nhật trên lifeline biểu thị khoảng thời gian một đối tượng đang thực hiện một tác vụ hoặc xử lý thông điệp.
    Thể hiện: Object đang xử lý logic.
    Hiểu đơn giản: “Đang chạy code”.
  4. Message (Thông điệp)
    Là phần QUAN TRỌNG NHẤT.
    Các loại message:
    • Call Message (Synchronous): Gọi hàm và chờ kết quả. Ví dụ: userService.login()
    • Asynchronous Message: Gọi nhưng không chờ. Dùng trong: Event-driven, Message queue.
    • Return Message: Trả kết quả về caller.
    • Self Message: Object gọi chính nó.
    • Create Message: Tạo object mới.
    • Destroy Message: Hủy object.
    • Recursive Message: Gọi đệ quy.

    *Tất cả message đều là communication giữa lifelines.

  5. Note (Ghi chú)
    Thêm comment, không ảnh hưởng logic.

Cách đọc Sequence Diagram

Quy tắc đọc:

  • Bắt đầu từ trên xuống dưới
  • Theo thứ tự message
  • Xem: Ai gọi ai, Gọi cái gì, Khi nào

Ví dụ: User → LoginController → AuthService → Database

Combined Fragments

Dùng để mô tả logic phức tạp:

  • alt (if-else): Điều kiện rẽ nhánh
  • loop: Lặp
  • opt: Optional
  • par: Song song (parallel)

Ví dụ alt:

  • [valid] → login success
  • [invalid] → error

*alt = if/else trong code

Phân loại Sequence Diagram

  • System Sequence Diagram (SSD):
    Chỉ có: Actor, System (black box). Dùng trong phân tích requirement.
  • Detailed Sequence Diagram:
    Có: Class, Object, Method call. Dùng trong design/code.

Quy trình xây dựng cơ bản

Để tạo một Sequence Diagram hiệu quả, bạn có thể thực hiện theo các bước sau:

  1. Xác định phạm vi: Chọn một kịch bản hoặc chức năng cụ thể cần mô hình hóa (ví dụ: đăng nhập, thanh toán).
  2. Liệt kê đối tượng: Xác định tất cả tác nhân và các đối tượng/module hệ thống tham gia vào kịch bản đó.
  3. Phác thảo trình tự: Vẽ các thông điệp theo thứ tự thời gian (từ trên xuống dưới), bắt đầu từ tác nhân kích hoạt.
  4. Bổ sung chi tiết: Thêm các hộp kích hoạt, điều kiện (nếu có) và phản hồi để hoàn thiện luồng xử lý.

Khi nào nên dùng Sequence Diagram?

Nên dùng khi:

  • Mô tả use case
  • Thiết kế API flow
  • Debug logic
  • Làm tài liệu cho team

Không nên dùng khi:

  • Mô tả structure → dùng Class Diagram
  • Mô tả trạng thái → dùng State Diagram
