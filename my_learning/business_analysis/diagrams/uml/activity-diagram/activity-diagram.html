Activity Diagram

Activity Diagram là một loại biểu đồ thuộc mô hình động (Dynamic Model) trong UML (Unified Modeling Language) dùng để mô tả:

  • Luồng công việc (workflow)
  • Quy trình nghiệp vụ (business process)
  • Luồng xử lý trong hệ thống
  • Các hoạt động tuần tự + song song + rẽ nhánh

Hiểu đơn giản:
Nó giống như flowchart nâng cấp (có thêm xử lý song song & logic phức tạp)

Activity Diagram tập trung vào luồng điều khiển từ hoạt động này sang hoạt động khác, thể hiện trình tự hành động để đạt mục tiêu cụ thể. Nó dùng để mô hình hóa workflow giữa use case, hoạt động phức tạp trong operation, hoặc quy trình kinh doanh có phối hợp song song. Khác flowchart thông thường, nó hỗ trợ phân nhánh, hợp nhất và hoạt động đồng thời.

Mục đích & Ý nghĩa

Mục đích chính

  • Mô tả quy trình nghiệp vụ
  • Mô hình hóa logic xử lý
  • Phân tích hệ thống trước khi code

Lợi ích

  • Hiểu toàn bộ flow hệ thống
  • Phát hiện bottleneck (điểm nghẽn)
  • Giao tiếp giữa BA – Dev – Stakeholder
  • Làm nền cho Sequence Diagram

Ví dụ:

  • Flow đăng nhập
  • Quy trình đặt hàng
  • Luồng xử lý thanh toán

Thành phần Chính

Activity Diagram gồm các ký hiệu cơ bản sau:

Thành phần Mô tả Ký hiệu UML
Initial Node (Nút bắt đầu) Điểm khởi đầu quy trình. Hình tròn đen
Activity/Action Hành động hoặc tác vụ cụ thể đại diện cho 1 bước xử lý. Activity là quy trình lớn, Action là bước nhỏ trong Activity Hình chữ nhật bo góc
Control Flow (Luồng điều khiển) Xác định thứ tự thực hiện, có thể kèm điều kiện Mũi tên
Object Flow Luồng đối tượng giữa các activity Mũi tên đứt nét
Decision Node (Nút quyết định/Branch) Phân nhánh dựa trên điều kiện (guard) Hình thoi
Merge Node Hợp nhất các nhánh quyết định Hình thoi
Fork Node Phân nhánh thành nhiều luồng song song Thanh ngang dày
Join Node Hợp nhất luồng song song (chờ tất cả hoàn thành) Thanh ngang dày
Final Node (Nút kết thúc) Kết thúc quy trình Hình tròn có chấm đen bên trong
Swimlane/Partition Chia diagram theo actor / role Hàng dọc/cột
Condition (Điều kiện) Điều kiện [valid]/[invalid]

Quy tắc Vẽ

  • Bắt đầu từ Initial Node, kết thúc tại Final Node.
  • Sử dụng Decision cho điều kiện (ví dụ: [đúng/sai]), Merge để hội tụ nhánh.
  • Fork/Join phải cân bằng số nhánh; Swimlane phân chia trách nhiệm rõ ràng.
  • Tránh vòng lặp vô hạn; ưu tiên luồng chính, chi tiết hóa use case nếu cần.

Các loại luồng trong Activity Diagram

  1. Sequential Flow (Tuần tự)
    A → B → C
  2. Branching (Rẽ nhánh)
    A → (if/else)
  3. Parallel Flow (Song song)
    A → (Fork) → B & C cùng chạy
  4. Loop (Lặp)
    quay lại bước trước
  5. Object Flow (luồng dữ liệu)
    truyền data giữa các activity

Cách xây dựng Activity Diagram

Bước 1: Xác định Use Case
Flow nào cần mô tả?

Bước 2: Xác định Start / End

Bước 3: Liệt kê Activities

Bước 4: Xác định:

  • Decision
  • Parallel
  • Loop

Bước 5: Thêm Swimlane (nếu cần)

Bước 6: Validate với Stakeholder

Khi nào dùng Activity Diagram?

Nên dùng khi:

  • Flow phức tạp
  • Có nhiều điều kiện
  • Có xử lý song song
  • BA cần mô tả nghiệp vụ

Không nên dùng khi:

  • Chỉ cần logic đơn giản → dùng flowchart
  • Muốn mô tả interaction → dùng Sequence Diagram

Nó hỗ trợ thiết kế Sequence Diagram sau và hiểu rõ hệ thống động.
