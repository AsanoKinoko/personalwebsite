Class Diagram

Class Diagram (Biểu đồ lớp) là một loại UML Structure Diagram (biểu đồ cấu trúc) dùng để mô tả cấu trúc tĩnh của hệ thống hướng đối tượng, tức là “hệ thống gồm những lớp nào, mỗi lớp có gì, và chúng liên hệ ra sao”.

Nó thể hiện:

  • Các class (lớp)
  • Attributes (thuộc tính)
  • Methods / Operations (phương thức)
  • Relationships (mối quan hệ giữa các lớp)

Hiểu đơn giản: Class Diagram = “bản thiết kế (blueprint)” của hệ thống OOP.

Vai trò & Mục đích sử dụng

Class Diagram có vai trò cực kỳ quan trọng giúp nhóm phát triển hiểu hệ thống ở mức cấu trúc, thống nhất thuật ngữ miền nghiệp vụ, và làm nền cho thiết kế mã nguồn:

  1. Trong phân tích & thiết kế
    • Mô hình hóa hệ thống trước khi code
    • Hiểu rõ cấu trúc dữ liệu và logic
  2. Trong phát triển phần mềm
    • Thiết kế kiến trúc lớp trước khi code
    • Là nền tảng của Object-Oriented Programming
    • Mapping trực tiếp sang code (Java, C#, etc.)
  3. Trong giao tiếp team
    • Giúp BA, Dev, Tester hiểu chung một ngôn ngữ
  4. Trong tối ưu hệ thống
    • Phát hiện lỗi design sớm (dependency, coupling…)

Thành phần của Class Diagram

1. Class (Lớp): là khối xây dựng cơ bản của class diagram, đại diện cho một nhóm đối tượng có cùng vai trò trong hệ thống.

Một class được biểu diễn bằng hình chữ nhật gồm 3 phần:

  • Class Name
  • attributes
  • methods

Ví dụ:

User
----------------
- id: int
- name: String
----------------
+ login()
+ logout()

2. Attributes (Thuộc tính)

  • Là dữ liệu của class (state)
  • Đại diện cho “class biết gì”

Trong diagram, thuộc tính thường ghi theo dạng visibility name: type. Ví dụ: age: int, name: String

3. Methods (Phương thức)

  • Là hành vi của class (behavior)
  • Đại diện cho “class làm gì”

Trong diagram, phương thức thường ghi dạng tenHam(thamSo: Kieu): KieuTraVe

Ví dụ:

+ getName(): String
+ setAge(age: int): void

4. Visibility (Phạm vi truy cập)

  • +: Public
  • -: Private
  • #: Protected
  • ~: package

Các mối quan hệ (Relationships)

Loại quan hệ Mô tả Ví dụ Ghi chú
Association (Liên kết) Quan hệ cơ bản giữa 2 class, nó biểu thị rằng các đối tượng của hai lớp có liên hệ với nhau. User - Order Có thể: 1-1, 1-N, N-N
Dependency (Phụ thuộc) Class A “dùng tạm thời” class B, nghĩa là thay đổi ở lớp B có thể ảnh hưởng lớp A. Quan hệ yếu nhất. PaymentService → Logger Mũi tên đứt nét
Aggregation (Kết tập) Quan hệ whole-part (yếu). Thành phần có thể tồn tại độc lập với toàn thể. Team ◊------ Player Dùng hình thoi rỗng
Composition (Hợp thành) Quan hệ whole-part (mạnh), là aggregation chặt. Part không thể tồn tại nếu thiếu whole, nghĩa là khi “whole” bị hủy thì “part” cũng bị hủy theo House ♦---- Room Dùng hình thoi đặc
Generalization/Inheritance (Kế thừa) Biểu thị quan hệ “is-a”. Lớp con kế thừa thuộc tính và hành vi của lớp cha. Lớp con → Lớp cha Mũi tên tam giác rỗng hướng về lớp cha
Realization (Triển khai Interface) Class implement interface. Class Bird implements Flyable Mũi tên đứt nét + tam giác rỗng hướng về interface

Tổng quan độ mạnh quan hệ:

Dependency < Association < Aggregation < Composition < Inheritance < Realization

Các thành phần mở rộng

  1. Interface
    • Chứa các method abstract
    • Class phải implement
  2. Abstract Class
    • Không thể khởi tạo object
    • Có thể chứa method abstract + concrete
  3. Enum
    • Tập giá trị cố định. Ví dụ: Status = {Pending, Done, Cancelled}
  4. Association Class
    • Quan hệ có thuộc tính riêng.
    • Ví dụ: Student --- Course (với Enrollment(score, date) ở giữa)
  5. Multiplicity
    • Cho biết một đối tượng của lớp này có thể liên kết với bao nhiêu đối tượng của lớp kia.
    • Các ký hiệu phổ biến là 1, 0..1, * hoặc 0..*, 1..*, và các khoảng như 3..5.
    • Đây là phần rất quan trọng để mô tả đúng ràng buộc nghiệp vụ.
    • Ví dụ: 1 (đúng một), 0..1 (không hoặc một), * (nhiều), 1..* (ít nhất một).

Ví dụ Class Diagram đơn giản:

User
----------------
- id: int
- name: String
----------------
+ login()
+ logout()

Order
----------------
- orderId: int
- total: double
----------------
+ createOrder()

User 1 ---- * Order
(Nghĩa là: 1 User có nhiều Order)

Best Practices

  1. Đặt tên đúng chuẩn: Class → danh từ (User, Product), Method → động từ (create, update)
  2. Không vẽ quá chi tiết: Chỉ giữ phần cần thiết
  3. Tránh coupling cao: Dependency quá nhiều → design lỗi
  4. Phân nhóm rõ ràng: Module theo domain
  5. Đọc dễ hiểu > đầy đủ

Lỗi thường gặp

  • Đưa quá nhiều chi tiết cài đặt vào mô hình khái niệm.
  • Nhầm aggregation với composition.
  • Quên multiplicity.
  • Đặt tên lớp và phương thức không nhất quán với miền nghiệp vụ.
  • Nhồi quá nhiều lớp vào một sơ đồ duy nhất.

Một sơ đồ tốt thường đơn giản, có chủ đích rõ ràng, và chỉ chứa những gì cần cho mục tiêu hiện tại. Nếu mục tiêu là phân tích nghiệp vụ thì đừng sớm chốt chi tiết kỹ thuật quá sâu; nếu mục tiêu là thiết kế code thì cần đủ chi tiết để triển khai.

Khi nào dùng Class Diagram?

Class Diagram phù hợp nhất khi bạn cần mô tả cấu trúc tĩnh của miền bài toán hoặc thiết kế hướng đối tượng. Nó không phù hợp để mô tả luồng thời gian, hành vi theo trình tự, hay trạng thái thay đổi theo thời gian; những phần đó thường thuộc Sequence Diagram, Activity Diagram, hoặc State Diagram.

Bạn nên dùng khi:

  • Thiết kế database (ERD gần giống)
  • Thiết kế hệ thống OOP
  • Phân tích domain (DDD)
  • Chuẩn bị code backend (Spring, .NET, Node)
